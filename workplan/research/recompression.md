# Research: Quality-Preserving Recompression Options

## 1. Summary — Candidate Conversion Matrix

| Source format | Recommended target | Reversibility class | Expected savings | Confidence |
|---|---|---|---|---|
| Baseline JPEG (JFIF, YCbCr 8-bit) | JPEG XL, `--lossless_jpeg=1` (default) | **Bit-exact** — original JPEG byte stream reconstructable via `djxl --reconstruct_jpeg` (spec-level jbrd mechanism) | ~18–22% smaller (20% figure from JPEG XL core designers; Google Research cites 22%; libjxl marketing cites up to 30% at high effort) | High — primary specs + running code agree |
| Progressive / arithmetic-coded / CMYK / some non-4:2:0 YCbCr JPEGs | **Do not transcode losslessly** — either leave untouched or fall back to lossy re-encode (not quality-preserving) | N/A — `cjxl` fails outright or silently falls back to pixel-based lossy VarDCT | N/A | High — reproduced in multiple libjxl issues |
| PNG (any bit depth) | JPEG XL, Modular mode, `-d 0` (mathematically lossless) | **Pixel-exact** (not byte-exact — this re-encodes, it does not preserve the original PNG bytes) | ~35% smaller than optimized PNG for 8-bit; larger gains at higher bit depth (Google Research) | High for pixels; medium for metadata/ancillary-chunk fidelity (see §4) |
| PNG (any bit depth) | Lossless WebP (`cwebp -lossless`) | Pixel-exact, not byte-exact | ~26–42% smaller than PNG depending on baseline (libpng vs. ZopfliPNG) | High — Google's own study |
| PNG | Lossless AVIF | Pixel-exact, not byte-exact | **Not recommended** — real-world reports show lossless AVIF often *larger* than optimized PNG or even a zipped BMP | High (reproduced maintainer-acknowledged issue) |
| TIFF (uncompressed or PackBits) | TIFF with LZW or Deflate/ZIP compression (same container) | **Byte-different, pixel-exact**, fully standard or re-encode as JPEG XL Modular lossless | LZW: 30–90% depending on content; Deflate: ~5–10% better than LZW | High — TIFF 6.0 spec is unambiguous about lossless codecs |
| BMP | PNG (or JXL Modular lossless) | Pixel-exact | Large (BMP is normally uncompressed) — savings dominated by target codec, not BMP's own RLE | High |
| Camera RAW (CR2/NEF/ARW/DNG etc.) | **Never convert** | N/A | N/A | High — see §5 |
| Any file with unrecognized/unsafe-to-copy ancillary chunks or unknown boxes | **Never convert** unless the tool proves round-trip preservation of that data | N/A | N/A | High — spec-mandated caution (see §5) |

**Bottom line for LensLocker's managed store:**
1. JPEG → JPEG XL lossless transcode is the only conversion in this survey with a *spec-backed, byte-exact* reversibility guarantee, and only for a subset of real-world JPEGs (baseline, standard chroma subsampling). It should be gated by a round-trip verification (transcode, reconstruct, byte-compare) before the original is deleted, and files that fail transcoding must be left as-is.
2. PNG → JXL Modular lossless or → lossless WebP are legitimate space savers but are **not byte-exact**; they are pixel-exact re-encodes. Ancillary PNG data (unknown chunks, unusual color types, embedded profiles beyond ICC) is at risk unless explicitly carried over by the encoder, so PNGs with unrecognized ancillary chunks should be excluded.
3. Lossless AVIF is not competitive for lossless still-image compression and should not be used as a lossless target.
4. TIFF and BMP have native (or trivial-target) lossless recompression paths; no need to introduce JPEG XL there unless simplicity of a single managed format is prioritized over minimizing tooling surface.
5. Camera RAW and any file type LensLocker cannot fully parse (unknown chunks/boxes, proprietary MakerNotes) must never be converted.

## 2. JPEG → JPEG XL Lossless Transcode

**Mechanism.** JPEG XL's coding tools are a superset of legacy JPEG's: the VarDCT mode can directly consume the source JPEG's existing DCT coefficients into 8×8 blocks and re-entropy-code them more efficiently, rather than decoding to pixels and re-encoding. Because generating a JPEG bitstream from a given set of DCT coefficients is not unique (padding, restart markers, Huffman table choices, chroma subsampling exact layout can vary), libjxl stores extra bookkeeping — the JPEG Bitstream Reconstruction Data — in a `jbrd` box inside the JXL container so the *original JPEG byte stream* can be regenerated bit-for-bit, not merely the pixel data. Source: [libjxl format_overview.md](https://github.com/libjxl/libjxl/blob/main/doc/format_overview.md) — "the `jbrd` box ... contains information needed for bit-identical reconstruction of the original JPEG file... it is only needed to reconstruct the original JPEG file[, not] to display a recompressed JPEG image."

**Is bit-exact reconstruction guaranteed by the spec?** The JPEG XL core-team paper states plainly: "A unique feature of JPEG XL is its ability to losslessly recompress existing JPEG images ... while allowing byte-for-byte exact reconstruction of the original JPEG file. This is possible because the coding tools of JPEG XL are a superset of the tools available in JPEG." Source: [The JPEG XL Image Coding System (arXiv 2506.05987)](https://arxiv.org/html/2506.05987), authored by the format's designers. The JPEG committee's own site echoes this: "Existing JPEG files can be losslessly transcoded to JPEG XL... These can be restored into the exact same JPEG file, ensuring backward compatibility with existing JPEG-based applications." Source: [jpeg.org/jpegxl](https://jpeg.org/jpegxl/).

Practically, this is *not unconditional* — it only applies to a subset of real-world JPEGs:
- Works reliably for standard **baseline** JPEGs.
- Progressive-scan JPEGs: mixed results — combining transcode with cjxl's `-p` progressive-*output* flag causes it to silently fall back to lossy VarDCT re-encoding rather than lossless transcode ([GitLab libjxl issue #133](https://gitlab.com/wg1/jpeg-xl/-/issues/133)); progressive *input* JPEGs are not guaranteed to transcode.
- Arithmetic-coded and CMYK JPEGs frequently fail outright with "unsupported type" errors from cjxl.
- Even some ordinary YCbCr baseline JPEGs (e.g., one produced by WhatsApp recompression, 4:2:0 chroma) have been reported to fail with the same "unsupported type (e.g. CMYK)" error, indicating the supported-subset boundary is narrower/fuzzier than the format's marketing suggests. Source: [libjxl issue #3604](https://github.com/libjxl/libjxl/issues/3604) (open, root cause not fully diagnosed in the thread).

**Practical implication for LensLocker:** treat lossless-JPEG-transcode as "attempt and verify," not "assume." The safe pattern is: run `cjxl --lossless_jpeg=1`, then verify with `djxl --reconstruct_jpeg` (or the encoder's own internal reconstruction check) and byte-compare against the original before deleting the source JPEG. If reconstruction fails or isn't attempted, do not delete/replace the original.

**Space savings.** ~20% is the figure consistently cited by primary/near-primary sources:
- JPEG XL designers' paper: "~20%" (arXiv 2506.05987).
- Google Research: "reduce ... storage costs of JPEG by 22%" ([Benchmarking JPEG XL lossy/lossless image compression](https://research.google/pubs/benchmarking-jpeg-xl-lossylossless-image-compression/)).
- cjxl documentation/community sources describe up to ~30% at higher encode effort settings ([cjxl man page](https://manpages.debian.org/unstable/libjxl-tools/cjxl.1.en.html) and related discussion).

**Encoder/decoder maturity.** libjxl (the C++ reference implementation providing `cjxl`/`djxl`) is still pre-1.0: latest tag as of this research is **v0.12.0** (July 1, 2026); the project has not committed to a 1.0/stability milestone. Source: [libjxl releases](https://github.com/libjxl/libjxl/releases). Browser support is reappearing but is not yet default-on anywhere: Chrome (v145, Feb 2026) ships a new pure-Rust decoder (`jxl-rs`) behind `chrome://flags/#enable-jxl-image-format`; Firefox 152 (June 2026) has it behind a disabled-by-default pref, with several blockers (color management, progressive decoding, HDR, WPT conformance) before stable release. Source: search of [caniuse.com/jpegxl](https://caniuse.com/jpegxl) and vendor release notes (secondary aggregation; browser-vendor bug trackers are the primary sources but were not individually fetched here — flagged as a gap).

**Rust story.**
- **jxl-oxide** ([crates.io](https://crates.io/crates/jxl-oxide), [GitHub](https://github.com/tirr-c/jxl-oxide)) — pure-Rust, **decode-only** JPEG XL implementation, v0.12.6, actively maintained (697 commits, 454 stars as observed). Has ICC handling via optional `lcms2` or pure-Rust `moxcms` backends. No JPEG-bitstream-reconstruction (jbrd) support surfaced in its documented API — it decodes to pixels, so it cannot be used alone to recover the original JPEG bytes.
- **jpegxl-rs** ([crates.io](https://crates.io/crates/jpegxl-rs), [GitHub](https://github.com/inflation/jpegxl-rs)) — safe FFI wrapper around the C++ libjxl reference implementation. Supports both encode and decode, explicitly supports "reconstruct JPEG, fallback to pixels if JPEG reconstruction is not possible" — i.e., it exposes the jbrd bit-exact reconstruction path. Current release 0.15.0 (paired with libjxl 0.12.0, dated July 3, 2026). **License: GPL-3.0-or-later** — a copyleft dependency that needs explicit sign-off given LensLocker's license posture (flagged as open question, §6).
- **libjxl/jxl-rs** ([GitHub](https://github.com/libjxl/jxl-rs)) — the official pure-Rust reimplementation being built by the libjxl team itself (this is the decoder Chrome/Firefox are integrating). **Decode-only, explicitly work-in-progress** ("aims to decode all conformant JPEG XL bitstreams correctly," invites bug reports). Not encode-capable, not yet declared production-ready.
- **cjxl / djxl CLI** — shipped as part of libjxl; can be shelled out to from Rust as a subprocess if avoiding GPL linkage or unstable Rust bindings is preferred. This sidesteps the jpegxl-rs GPL question but adds a runtime dependency on the libjxl binaries and their own pre-1.0 stability caveats.

**Net assessment:** For a *decoder* embedded in the app (to display .jxl or verify reconstructions), jxl-oxide is the most Rust-idiomatic, permissively-licensed option, but it cannot itself do JPEG-bitstream reconstruction verification. For an *encoder* with reconstruct-JPEG capability, the realistic choices are (a) jpegxl-rs (GPL-3.0-or-later, FFI to C++), or (b) shelling out to the `cjxl`/`djxl` CLI binaries. Neither is a lightweight, pure-Rust, permissively-licensed, encode+reconstruct solution today.

## 3. PNG → Lossless WebP / Lossless JXL / Lossless AVIF

**PNG → Lossless WebP.** Google's own compression study: "WebP lossless compression is ... 42% better than libpng ... 23% better than ZopfliPNG," with maximum settings reducing average bits-per-pixel from 5.05 (ZopfliPNG) to 3.96 across their web corpus, and WebP beating size-optimized PNGs on "over 99%" of tested images. Source: [WebP Compression Study](https://developers.google.com/speed/webp/docs/webp_study) / [Lossless and Transparency Encoding in WebP](https://developers.google.com/speed/webp/docs/webp_lossless_alpha_study). Decode support is universal in modern browsers and OS frameworks (this claim is uncited here — flagged as a gap; not independently verified against a primary compatibility table in this pass).

**PNG → Lossless JXL (Modular mode).** Google Research states lossless JPEG XL is "around 35% smaller than (optimized) PNG for 8-bit images, and the improvement is even larger for images with a higher bit depth." Source: [Benchmarking JPEG XL lossy/lossless image compression](https://research.google/pubs/benchmarking-jpeg-xl-lossylossless-image-compression/). This is the best pure compression ratio of the three lossless targets surveyed, at the cost of the encoder/decoder maturity caveats in §2.

**PNG → Lossless AVIF.** **Not recommended.** AVIF's predictive-transform lossless mode is not competitive: a maintainer-visible bug report shows a lossless AVIF encode (412 kB) losing to both a zipped BMP (406 kB) and losing badly to lossless WebP and lossless JXL (both under 250 kB) on the same source image. Source: [libavif issue #2434, "Lossless avif worse than a bmp in a zip file"](https://github.com/AOMediaCodec/libavif/issues/2434). AVIF's lossless mode exists (`avifenc -l`) per the [libavif avifenc man page](https://github.com/AOMediaCodec/libavif/blob/main/doc/avifenc.1.md) but is not a space-saving proposition versus PNG/WebP/JXL for typical photographic or graphic content.

**Rust encoder note (bears on AVIF viability generally, not just lossless):** the most mature pure-Rust AVIF encoder, `ravif` (used by `cavif-rs`, and pulled into the `image` crate), explicitly **does not implement lossless mode at all** — "setting quality to 100 just gives unreasonably bloated files" because its underlying `rav1e` AV1 encoder has never implemented AV1's lossless coding path (feature request open since 2018). Source: [rav1e issue #151, "Support lossless mode!"](https://github.com/xiph/rav1e/issues/151), [ravif on lib.rs](https://lib.rs/crates/ravif). This reinforces that AVIF should be excluded from the lossless-conversion candidate set for a pure-Rust implementation, independent of the compression-ratio argument above.

**Rust WebP encoder note:** `image-webp` ([GitHub](https://github.com/image-rs/image-webp), the `image` crate's pure-Rust WebP backend) currently **only implements lossless encoding** (no lossy), is pure-safe-Rust with no unsafe code, and per its own docs "often produces smaller files than PNG, even when compared against the slowest/highest compression options of PNG encoders" — though its compression ratio trails libwebp's own encoder since the implementation is deliberately simple/fast. This makes it an attractive, permissively-licensed, pure-Rust option specifically for the PNG→lossless-WebP conversion path.

## 4. TIFF / BMP Recompression

**TIFF.** The TIFF 6.0 specification (Adobe) defines three standard compression schemes still in wide use: LZW, Deflate ("ZIP"), and JPEG (lossy, out of scope here). Both LZW and Deflate are lossless. Typical LZW savings: 30–50% for photographic content, 50–70% for images with large solid-color regions, 70–90% for line art/text; Deflate/ZIP typically beats LZW by an additional 5–10% but is slower to encode (decode speed is comparable). Caveat: LZW's dictionary approach can *increase* file size on high-entropy 16-bit-per-channel TIFFs. Source: aggregated secondary summaries of the TIFF 6.0 spec's compression tags — the primary Adobe TIFF 6.0 PDF itself was not successfully fetched in this pass (network/cert error); flagged as a gap, should be verified against the primary spec PDF before implementation. TIFF can alternatively be re-encoded losslessly into JPEG XL Modular mode for a single managed-format story, at the cost of introducing JXL's maturity caveats to a wider set of files.

**BMP.** BMP is conventionally stored uncompressed; an RLE variant exists for 4-/8-bit palettized images (Microsoft's BI_RLE4/BI_RLE8), which is lossless but only helps on flat/indexed content and can *increase* size on photographic data. There is no widely-supported "recompress BMP as BMP" story worth pursuing — the only sensible action is re-encoding losslessly into a real lossless container (PNG, or JXL Modular) rather than trying to shrink the BMP container itself.

## 5. What Must Never Be Converted

**Camera RAW (CR2, NEF, ARW, proprietary and DNG).** Raw files hold sensor-level data captured before in-camera processing (no demosaicing, white balance, or sharpening baked in). Two independent risk classes:
1. **Undocumented proprietary structure.** Most camera manufacturers' raw formats are not public specifications; even where a raw file is TIFF-structured, it typically carries proprietary MakerNote blocks that generic tooling cannot safely parse, preserve, or reproduce byte-for-byte. Re-encoding risks silently dropping or corrupting data no re-encoder can enumerate. Source: general characterization from [Adobe DNG documentation](https://helpx.adobe.com/camera-raw/digital-negative.html) and surrounding Adobe DNG converter documentation, contrasting DNG's openly published spec against "many different raw formats, the specifications for which are not publicly available."
2. **Even DNG, the "open" raw container, is not a safe re-encode target for a generic pipeline** — a correct DNG converter needs camera-model-specific knowledge to interpret the source raw data correctly; that's a fundamentally different (and much riskier) operation than a well-defined bit-exact or pixel-exact transcode between two fully specified formats like JPEG→JXL or PNG→WebP.
- **Recommendation:** camera RAW files should never be touched by the managed store's recompression logic — copy/store verbatim.

**Files with unknown or unsafe-to-copy ancillary data.** The PNG specification formally defines this risk class: a "PNG editor" is expected to preserve ancillary chunks it doesn't understand, and the spec distinguishes "safe-to-copy" ancillary chunks (may be freely retained by an editor that doesn't understand them, provided ordering constraints relative to IDAT are respected) from "unsafe-to-copy" ones (must not be blindly carried over by a program that doesn't understand their semantics, because they may depend on image data the editor is about to change). Source: [PNG Specification: Chunk Specifications, W3C](https://www.w3.org/TR/PNG-Chunks.html) / [PNG Specification: Chunk Ordering Rules](https://www.w3.org/TR/PNG-Ordering.html). Any converter that does not explicitly enumerate and re-emit every chunk type present in a source PNG (or the equivalent box/tag concept in other containers) is, by the spec's own definition, not a safe general-purpose PNG editor.
- **Recommendation:** before converting a PNG (or any chunk/box-based container), the pipeline must inventory all chunks/boxes present; if any are outside an explicitly-supported allowlist (IHDR/PLTE/IDAT/IEND/tRNS/gAMA/cHRM/sRGB/iCCP/tEXt/zTXt/iTXt/eXIf/bKGD/pHYs — standard ancillary chunks with known semantics), the file should be excluded from lossless recompression rather than risk silently dropping unsafe-to-copy data.

## 6. Metadata (EXIF/XMP/ICC) Round-Trip Survival

| Conversion | EXIF | XMP | ICC | Tooling notes / source |
|---|---|---|---|---|
| JPEG → JXL lossless transcode | Preserved | Preserved | Preserved | When `--allow_jpeg_reconstruction=1` (default), full JPEG bytes — including all embedded metadata — are recoverable via reconstruction. libjxl also stores Exif/XMP in dedicated `Exif`/`xml ` boxes for direct (non-reconstructed) access; codestream orientation data takes precedence over any conflicting box-level Exif orientation. Note: `cjxl` explicitly documents that stripping metadata during lossless JPEG recompression **requires disabling reconstruction** (`--allow_jpeg_reconstruction=0`) — i.e., you cannot both keep byte-exact reconstruction and strip metadata. Source: [libjxl format_overview.md](https://github.com/libjxl/libjxl/blob/main/doc/format_overview.md), [cjxl man page](https://manpages.debian.org/unstable/libjxl-tools/cjxl.1.en.html). |
| PNG → JXL Modular lossless | Carries over if encoder reads PNG `eXIf` chunk | Carries over if encoder reads PNG XMP (`iTXt`, keyword `XML:com.adobe.xmp`) | Carries over if encoder reads PNG `iCCP` | Not independently verified against `cjxl` source/tests in this pass. There are open libjxl issues around ICC handling being lossy/buggy in some pipelines (e.g., alpha-channel removal dropping the profile) — treat ICC preservation as "usually works, verify per-file," not guaranteed. Source: [libjxl issue #4164](https://github.com/libjxl/libjxl/issues/4164), [libjxl issue #2550](https://github.com/libjxl/libjxl/issues/2550). |
| PNG → Lossless WebP | Opt-in | Opt-in | Opt-in | `cwebp -metadata all` (or `-metadata exif,icc,xmp`) explicitly copies these from the source; default is `-metadata none`, i.e. **metadata is stripped unless requested**. Source: [cwebp man page, Debian](https://manpages.debian.org/testing/webp/cwebp.1.en.html). WebP's RIFF container has dedicated `EXIF`, `XMP `, and `ICCP` chunks (at most one of each) per the [WebP Container Specification](https://developers.google.com/speed/webp/docs/riff_container). |
| PNG/JPEG → AVIF | Opt-in | Opt-in | Opt-in | `avifenc --exif FILE` / `--xmp FILE` / `--icc FILE` attach metadata explicitly; corresponding `--ignore-exif` / `--ignore-xmp` / `--ignore-icc` flags exist. Default behavior (whether metadata is auto-carried from a JPEG/PNG source without flags) was not conclusively pinned down from the man page excerpt fetched — flagged as a gap. Source: [libavif avifenc.1.md](https://github.com/AOMediaCodec/libavif/blob/main/doc/avifenc.1.md). AVIF's underlying HEIF/MIAF container supports multiple EXIF and XMP blobs plus a `colr` box for ICC — structurally richer than WebP's "at most one each," per [av1-avif spec discussion](https://github.com/AOMediaCodec/av1-avif/issues/193). Moot given §3's recommendation against lossless AVIF. |
| TIFF → TIFF (LZW/Deflate) | Native (EXIF IFD pointer tag) | Native (tag 700) | Native (ICC profile tag 34675) | TIFF's own tag space already has first-class slots for all three; recompressing image strips/tiles in place doesn't touch the tag directory, so this is the lowest-risk metadata path of everything surveyed — not independently re-verified against the primary TIFF 6.0 PDF in this pass (see §4 gap). |

**General pattern:** metadata survival is never "free" — every tool surveyed treats EXIF/XMP/ICC as either (a) bundled automatically only when doing full byte-exact reconstruction (JPEG→JXL path), or (b) an explicit opt-in flag the caller must set (`cwebp -metadata`, `avifenc --exif/--xmp/--icc`). A naive `cjxl in.jpg`, `cwebp in.png`, or `avifenc in.png` invocation without the right flags can silently drop metadata even though the pixels/bytes are otherwise faithfully converted. **LensLocker's pipeline must explicitly request full metadata preservation on every conversion path and verify presence post-conversion — never rely on tool defaults.**

## 7. Open Questions

1. **jpegxl-rs is GPL-3.0-or-later.** If LensLocker ships as closed-source or under a non-copyleft license, using jpegxl-rs (or linking libjxl directly) needs explicit legal review; the alternative (shelling out to the `cjxl`/`djxl` CLI as a separate process) avoids linkage but adds a packaging/distribution dependency on those binaries and their own update cadence.
2. **No pure-Rust, permissively-licensed encoder exists today that both losslessly transcodes JPEG→JXL *and* supports jbrd-based bit-exact reconstruction.** jxl-oxide (decode-only) and libjxl/jxl-rs (decode-only, WIP) don't cover this; only jpegxl-rs (GPL, FFI) or the CLI binaries do. This is a real gap for a "100% offline, self-contained Rust binary" design goal and should be revisited as jxl-rs matures (it's the same codebase browsers are adopting, so it has strong long-term backing) or resolved via subprocess-CLI packaging.
3. **libjxl is pre-1.0 (v0.12.0 as of July 2026).** No formal stability/compatibility guarantee has been made by the project for its bitstream, CLI flags, or reconstruction correctness across versions. A version pin plus the byte-exact round-trip verification recommended in §2 mitigates but does not eliminate this risk.
4. **JPEG lossless-transcode "supported subset" boundary is fuzzy in practice** (§2, libjxl issue #3604 shows an apparently-ordinary YCbCr JPEG failing). The safe engineering answer — attempt, verify by reconstruction, keep the original on any failure — sidesteps needing a precise definition of "supported," but it does mean a non-trivial fraction of real-world JPEGs (exact rate unmeasured here) will not be eligible for this conversion at all.
5. **PNG ancillary-chunk allowlisting (§5) needs to be validated against whatever PNG encoder/decoder crate LensLocker chooses** — does it expose enough low-level chunk visibility to implement the "inventory before convert" gate, or would that require a second, chunk-aware PNG parser purely for this safety check?
6. **TIFF primary-spec claims in §4 were not verified against the actual Adobe TIFF 6.0 PDF** (fetch failed on a TLS certificate error mid-research) — the LZW/Deflate savings percentages there trace to secondary summaries, not the spec text itself. Worth a follow-up fetch before this becomes load-bearing for an implementation decision.
7. **Browser/decode-support claims for JPEG XL (§2) were pulled from aggregator/community sources (caniuse, vendor blogs), not individually verified against Chromium/Firefox bug trackers** — irrelevant to LensLocker's *storage* correctness, but relevant if any exported/shared file is expected to open in a browser.
