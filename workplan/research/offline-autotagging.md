# Research: fully-local ML auto-tagging + similarity search from Rust

**Ticket:** [016 — Survey offline auto-tagging options](../tickets/016-research-offline-autotagging.md)
**Status:** research only — informs spec forward-compatibility and schema reservations. No decision taken here.

## Summary — the 2-3 realistic stacks, and what the schema must reserve

**Stack A (recommended starting point): `ort` (ONNX Runtime) + an ONNX-exported SigLIP/CLIP checkpoint + `sqlite-vec` (brute-force) as a SQLite virtual table.**
`ort` is the most production-proven Rust inference path, its `directml` feature gives Windows GPU acceleration across NVIDIA/AMD/Intel/Qualcomm from one code path (no CUDA-only lock-in) ([DirectML EP docs](https://onnxruntime.ai/docs/execution-providers/DirectML-ExecutionProvider.html)), and Immich already publishes ready-to-use ONNX exports of CLIP/SigLIP/SigLIP2 checkpoints under its own Hugging Face org that a Rust app could adopt or re-export the same way ([immich-app HF org](https://huggingface.co/immich-app/ViT-B-16-SigLIP-256__webli)). `sqlite-vec` keeps embeddings inside the existing SQLite catalog rather than adding a second database — brute-force is its only algorithm today, but that is documented as adequate through roughly the "low hundreds of thousands" of vectors before an ANN index is needed ([sqlite-vec README](https://raw.githubusercontent.com/asg017/sqlite-vec/main/README.md), [sqlite-vec v0.1.0 announcement](https://alexgarcia.xyz/blog/2024/sqlite-vec-stable-release/index.html)), which covers the 100k+ target with headroom.

**Stack B (pure-Rust, no external runtime DLL): `candle` + `candle-transformers`' built-in CLIP/SigLIP implementations + local safetensors weights, still paired with `sqlite-vec` or a brute-force scan.** Avoids shipping/managing `onnxruntime.dll`, but candle's GPU acceleration on Windows is CUDA-only (no DirectML, no documented Windows GPU path) ([candle README](https://raw.githubusercontent.com/huggingface/candle/main/README.md)) — CPU-only would be the realistic default for a Windows-first app that must support AMD/Intel GPUs too.

**Stack C (fallback / longer runway): `burn` with its `wgpu` backend.** `burn`'s wgpu backend targets Vulkan/DirectX12/Metal, which is the only one of the three runtimes offering vendor-agnostic Windows GPU acceleration *without* an ONNX Runtime dependency ([burn-wgpu README](https://github.com/tracel-ai/burn/blob/main/crates/burn-wgpu/README.md), [Burn cross-platform GPU backend blog](https://burn.dev/blog/cross-platform-gpu-backend/)). Its ONNX import path (`burn-import`) is real but explicitly limited in operator coverage, and no prebuilt CLIP/SigLIP implementation exists yet — this stack means porting a model, not just loading one.

**What the schema/spec must reserve now, regardless of which stack wins later:**
- A place for **one embedding vector per image** (dimensionality varies by model: 512 for OpenAI CLIP ViT-B/32, 768 for CLIP ViT-L/14 and SigLIP base — the column/table must not hardcode a dimension) plus a **model identity + version tag** per embedding row, since re-tagging with a newer model must not silently mix incompatible vector spaces.
- Storage for **zero-shot tag labels/scores** distinct from the embedding itself (auto-tags are a derived, re-runnable projection of the embedding against a label set, not raw model output).
- A **content hash → embedding** relationship that survives file moves within the managed store (embeddings are expensive to recompute; they should key off the existing BLAKE3 identity, not file path).
- Headroom for a **sidecar ANN index file** next to the `.sqlite` catalog (HNSW crates persist as their own dump files, not SQLite rows) in case brute-force stops being adequate well past 100k images.
- A **license/provenance field for the bundled model weights** the installer ships, since redistribution terms differ by checkpoint (see below) and the app is offline-distributed, not downloaded per-user from Hugging Face at install time.

## 1. Runtimes

### `ort` (Rust bindings for Microsoft ONNX Runtime)
- Actively maintained; latest is `2.0.0-rc.12` (June 25, 2026) — [docs.rs/ort](https://docs.rs/ort/latest/ort/).
- Windows CPU execution is the default/baseline path; no special setup needed beyond the crate itself.
- **DirectML**: the `directml` Cargo feature enables the DirectML execution provider "for Windows x86/x64 targets with dedicated GPUs supporting DirectX 12" — [search-derived, ort docs](https://ort.pyke.io/setup/cargo-features). DirectML itself (Microsoft's own EP docs) supports NVIDIA Kepler+, AMD GCN-1st-gen+, Intel Haswell HD Graphics+, and Qualcomm Adreno 600+ — i.e. essentially any DX12-capable GPU, which is the only cross-vendor GPU story of the three runtimes on Windows — [DirectML EP docs](https://onnxruntime.ai/docs/execution-providers/DirectML-ExecutionProvider.html). CUDA and TensorRT execution providers also exist as separate features.
- **Binary size / linking**: `ort` supports three linking strategies via `ORT_STRATEGY` — `download` (default; fetches prebuilt ONNX Runtime binaries from Microsoft at *build* time, not app runtime), `system` (point at a locally built/installed copy), and `load-dynamic` (load `onnxruntime.dll` at runtime via `ORT_DYLIB_PATH`, recommended by the docs as "less troublesome") — [pykeio/ort linking docs, search-derived](https://github.com/pykeio/ort/blob/main/docs/content/setup/linking.mdx). The core CPU `onnxruntime.dll` for a Windows x64 release build is roughly **12 MB**; provider-specific DLLs (e.g. CUDA) can be much larger, but this wasn't confirmed for the DirectML-specific package size — flagged as unverified — [GitHub issue with file-size listing](https://github.com/microsoft/onnxruntime/issues/23077).
- Model loading: standard `.onnx` files via a `Session` API; once the runtime DLL and model file are bundled with the installer, no network access is needed at inference time (the `download` strategy only touches network at *build* time, not at app runtime).

### `candle` (huggingface/candle, pure-Rust framework)
- Builds on Windows CPU; documentation shows Windows-specific troubleshooting entries (e.g. linking errors on some `rustdoc`/`mdbook` paths) but no blocking Windows-CPU issue — [candle README](https://raw.githubusercontent.com/huggingface/candle/main/README.md).
- GPU: CUDA backend (multi-GPU via NCCL) and an Accelerate CPU-optimization path for Macs are documented; **no DirectML or other Windows-native GPU backend is mentioned anywhere in the README** — Windows GPU acceleration is effectively unavailable outside CUDA (NVIDIA-only) — [candle README](https://raw.githubusercontent.com/huggingface/candle/main/README.md).
- Binary size: pure Rust, no external runtime shared library to bundle (unlike `ort`); candle explicitly markets itself for "lightweight binaries" and serverless-style deployment — [candle README](https://raw.githubusercontent.com/huggingface/candle/main/README.md).
- Model loading: native `safetensors` loading (local files, no network needed), and `candle-transformers` already contains **CLIP**, **OpenCLIP**, and **SigLIP** model implementations (confirmed via the `candle-transformers/src/models/` directory listing and search corroboration) — [candle-transformers source tree](https://github.com/huggingface/candle/tree/main/candle-transformers/src/models), [search corroboration](https://huggingface.co/docs/transformers/main/en/model_doc/siglip).

### `burn` (tracel-ai/burn)
- Cross-platform by design; the project states "most backends support all operating systems" — [burn README, search-derived](https://raw.githubusercontent.com/tracel-ai/burn/main/README.md).
- Backends relevant to Windows GPU: **`wgpu`** targets Vulkan, DirectX11/12, Metal, OpenGL, and WebGPU from one backend — the only vendor-agnostic Windows GPU path among the three runtimes surveyed — [burn-wgpu README](https://github.com/tracel-ai/burn/blob/main/crates/burn-wgpu/README.md), [Burn cross-platform GPU backend blog](https://burn.dev/blog/cross-platform-gpu-backend/). A `spirv` feature further optimizes the Vulkan path (f16, TensorCores) over plain WGSL. CUDA and NdArray (CPU) backends also exist, and a LibTorch (`tch`) backend is available as an external dependency option.
- Binary size implications: not documented explicitly; the `wgpu` backend pulls in a shader-compilation stack (CubeCL) which is a real but unquantified size/complexity cost versus `candle`'s minimalism.
- Model loading: `burn-import` converts ONNX graphs into native Burn/Rust code, but "currently supports a limited set of ONNX operators" — [burn README, search-derived](https://raw.githubusercontent.com/tracel-ai/burn/main/README.md). No prebuilt CLIP/SigLIP implementation was found in Burn's model examples; using CLIP-family models here would mean porting, not loading.

**License verdicts:**
- OpenAI CLIP: **MIT (code, confirmed via repo LICENSE file)**; the Hugging Face model cards for the weights (`openai/clip-vit-base-patch32`, `-large-patch14`) carry **no explicit `license:` metadata tag** — ambiguous by omission, though understood to inherit the repo's MIT grant. The card's own "Out-of-Scope Use" section states *"Any deployed use case of the model – whether commercial or not – is currently out of scope,"* and marks surveillance/facial-recognition use as *"always out-of-scope"* — an ethics disclaimer from OpenAI's original paper, not a legal restriction inside the MIT grant itself, but worth a legal read given LumenVault is exactly this kind of deployed use — [openai/clip-vit-base-patch32](https://huggingface.co/openai/clip-vit-base-patch32).
- OpenCLIP/LAION checkpoints: **MIT for both code and weights**, and unlike OpenAI's cards, individual checkpoints (e.g. `laion/CLIP-ViT-B-32-laion2B-s34B-b79K`) do carry an explicit `license: mit` tag — cleaner provenance than OpenAI's untagged cards. The same inherited "deployed use out of scope" disclaimer language appears here too (see above) — again a caution, not a legal clause. Separately, the model card flags its LAION-2B training data as "uncurated" with potential for "disturbing content"; LAION's own FAQ clarifies the *dataset* itself ships no images (only URLs + alt-text) and rests on EU/German text-and-data-mining research exemptions rather than an explicit content license — this is about training-data provenance, not the trained *weights'* redistribution terms, but is the kind of provenance chain a legal reviewer would want documented — [laion/CLIP-ViT-B-32-laion2B-s34B-b79K](https://huggingface.co/laion/CLIP-ViT-B-32-laion2B-s34B-b79K), [LAION FAQ](https://laion.ai/faq/).
- SigLIP: **Apache-2.0 — permissive, requires attribution/NOTICE preservation**, confirmed via the HF API `license` field on both `google/siglip-base-patch16-224` and `google/siglip-so400m-patch14-384`; **no disclaimer/out-of-scope language was found on either card** — the cleanest license story of the three families surveyed.

**ONNX conversions:** Immich's own Hugging Face org (`immich-app`) publishes ready-made ONNX exports of multiple CLIP/SigLIP/SigLIP2 checkpoints (e.g. `immich-app/ViT-B-16-SigLIP-256__webli`, `immich-app/ViT-SO400M-16-SigLIP2-384__webli`), explicitly built "for use with Immich" — these are directly loadable by `ort`. The model cards for these conversions do not display their own separate license text distinct from the underlying model, but this was **not exhaustively confirmed for every checkpoint** — flagged as an open item — [immich-app ViT-B-16-SigLIP-256 model card](https://huggingface.co/immich-app/ViT-B-16-SigLIP-256__webli).

## 3. Embedding storage and search at 100k+ scale

- **`sqlite-vec`** (successor to `sqlite-vss`): a pure-C, dependency-free SQLite extension, "designed to run anywhere SQLite runs" including Windows and WASM; installable directly as a Rust crate (`cargo add sqlite-vec`), no network access required at runtime — [sqlite-vec GitHub](https://github.com/asg017/sqlite-vec). Dual-licensed **MIT/Apache-2.0** — [sqlite-vec GitHub](https://github.com/asg017/sqlite-vec). The `sqlite-vec` Rust crate itself is FFI bindings that register the C extension with `rusqlite` (`^0.31`), not a native Rust reimplementation — [docs.rs/sqlite-vec](https://docs.rs/sqlite-vec/latest/sqlite_vec/). Current status is pre-1.0 (v0.1.9, March 2026) with "expect breaking changes." It currently implements **brute-force/exhaustive search only** — no HNSW/IVF ANN index is shipped; the maintainer tracks ANN indexing as roadmapped for a future v0.2.0+ (GitHub issue #25), not yet shipped — [Alex Garcia's sqlite-vec blog](https://alexgarcia.xyz/blog/2024/sqlite-vec-stable-release/index.html).
- **Published brute-force benchmark (sqlite-vec's own numbers, not LumenVault-specific hardware/data):** at **100,000 vectors**, per-query KNN latency is **≤75ms at 768-dim**, ~105ms at 1536-dim, ~214ms at 3072-dim (float vectors; binary-quantized vectors are much faster, 11–124ms) — comfortably under a 200ms interactive-search target for CLIP/SigLIP-sized embeddings (512–768 dims), with headroom before the dimensionality where it would start to matter — [Alex Garcia's sqlite-vec blog](https://alexgarcia.xyz/blog/2024/sqlite-vec-stable-release/index.html). The author frames the extension as suited to "hundreds of thousands," not billions, of vectors — consistent with LumenVault's 100k+ target but a reason to re-benchmark before assuming it holds at 500k+.
- **Hand-rolled SIMD alternative:** `simsimd` (Apache-2.0/BSD-3-Clause, actively maintained, AVX2/AVX-512/NEON/SVE kernels) claims 10–200x speedup over naive cosine/dot-product/L2 implementations and is used internally by USearch; no independent 100k-vector Rust benchmark was found, but it's a viable fallback if `sqlite-vec`'s built-in brute force isn't fast enough — [docs.rs/simsimd](https://docs.rs/simsimd), [SimSIMD GitHub](https://github.com/ashvardanian/SimSIMD).
- **HNSW crates in Rust**, as a fallback if brute-force ever proves insufficient — all are embeddable in-process (no server) and would live as a sidecar index file next to the `.sqlite` catalog rather than inside SQLite itself:

  | Crate | License | Maintenance | Persistence |
  |---|---|---|---|
  | `hnsw_rs` | MIT/Apache-2.0 | Active (v0.3.4, Feb 2026) | Dump/reload to disk |
  | `usearch` | Apache-2.0 | Active (v2.26.0, Jul 2026); Rust bindings over C++ USearch, claims ~10x FAISS | File save/load |
  | `instant-distance` | MIT/Apache-2.0 | Stale (v0.6.1, Jun 2023) | Optional via `with-serde` |
  | `hora` | Apache-2.0 | Effectively dead (v0.1.1, Aug 2021) — not viable for new adoption | — |

  [lib.rs/hnsw_rs](https://lib.rs/crates/hnsw_rs), [lib.rs/usearch](https://lib.rs/crates/usearch), [lib.rs/instant-distance](https://lib.rs/crates/instant-distance), [lib.rs/hora](https://lib.rs/crates/hora)
- **Synthesis:** for LumenVault's 100k+ (not multi-million) target, `sqlite-vec`'s brute-force approach — kept inside the same SQLite catalog the app already uses, and backed by a real 100k-scale benchmark (≤75ms/query at 768-dim) — is both fast enough and the more architecturally consistent choice. `hnsw_rs` (pure Rust, actively maintained, persistable) is the concrete fallback crate to reserve schema/architecture room for if the library scales well past 100k toward the high hundreds-of-thousands or 1M+, where linear brute-force scan time would start eating into the interactive-latency budget.

## 4. Precedents

### Immich
- Runs ML as a **separate Python/FastAPI microservice**, isolated from the main NestJS server, packaged as its own Docker container — not in-process with the main app — [Immich ML service, search-derived from DeepWiki/GitHub](https://github.com/immich-app/immich/blob/main/machine-learning/README.md).
- Uses **CLIP-family embeddings for smart search** (multiple OpenCLIP/SigLIP/SigLIP2 checkpoints, self-published as ONNX exports under the `immich-app` HF org) and **InsightFace models** (`antelopev2`, `buffalo_l/m/s`) for facial recognition — [immich-app HF org](https://huggingface.co/immich-app/ViT-B-16-SigLIP-256__webli), [Immich ML README, search-derived](https://github.com/immich-app/immich/blob/main/machine-learning/README.md).
- Runtime is **ONNX Runtime**, with execution providers for CPU, CUDA, ROCm (AMD), OpenVINO (Intel), ARM NN, and RKNN — but its own hardware-acceleration docs explicitly state **"Only Linux and Windows (through WSL2) servers are supported"** for GPU acceleration — i.e. Immich has no native Windows GPU path (no DirectML) and requires WSL2 on Windows — [Immich ML hardware acceleration docs, search-derived](https://docs.immich.app/features/ml-hardware-acceleration).
- Resource footprint: official requirements call for **at least 6 GB RAM** for a smooth experience with ML enabled; the ML container can be disabled on 4 GB systems, and OpenVINO is separately documented to use more RAM than CPU-only inference — [Immich requirements, search-derived](https://docs.immich.app/install/requirements/), [Immich ML hardware acceleration docs](https://docs.immich.app/features/ml-hardware-acceleration).
- Embeddings are stored/searched in **PostgreSQL with pgvector**, not SQLite — a direct architectural contrast to LumenVault's SQLite-only commitment (confirmed via search corroboration of Immich's architecture; not independently re-verified against Immich's schema source in this pass).
- **Model provisioning is not fully offline out of the box**: models are **not baked into the Docker image** — they're pulled from Hugging Face into a model-cache volume on first use (confirmed by an open GitHub issue about the cache being ignored and models re-downloading) — [Immich issue #8237](https://github.com/immich-app/immich/issues/8237). This is an important caveat on Immich as a "fully local" existence proof: inference afterward is offline, but provisioning currently assumes at least one internet-connected first run (or a manual pre-download) — a constraint LumenVault must design around explicitly, since it ships weights inside the installer instead.
- **Face-recognition model license is a concrete redistribution-restriction example**: Immich's own re-hosted `buffalo_l` (InsightFace) checkpoint is tagged **"other"/InsightFace license** on its model card, which states permission was granted for Immich's specific project use but explicitly **"does not extend to redistribution or commercial use of their models by third parties"** — [immich-app/buffalo_l model card](https://huggingface.co/immich-app/buffalo_l/raw/main/README.md), [InsightFace license](https://github.com/deepinsight/insightface/tree/master/python-package#license). This is out of scope for LumenVault's CLIP/SigLIP embedding models (all MIT/Apache-2.0, cleanly redistributable — see §2), but it's a real-world precedent that not every ML model bundled by a local-first photo app is freely redistributable, and is directly relevant if/when a future face-grouping milestone considers InsightFace-style models.
- **Implication for LumenVault**: Immich is the closest existence proof of "ONNX Runtime + CLIP/SigLIP for local smart search," but its microservice/Postgres architecture is the opposite of LumenVault's single-binary, SQLite-centric, no-server design, and its model provisioning assumes network access at least once — it validates the *models and runtime*, not the *architecture* or the *zero-network provisioning story*.

### PhotoPrism
- ML runs **in-process** inside the Go binary (not a separate service) — TensorFlow via Go bindings historically, with a notable recent shift: face **detection** now uses an **ONNX Runtime-backed** model (SCRFD 0.5g) as of a 2026 release, while face **embedding** still uses **TensorFlow/FaceNet** (512-dim, L2-normalized vectors, clustered via DBSCAN) — [PhotoPrism face recognition docs](https://docs.photoprism.app/developer-guide/vision/face-recognition/).
- Image classification/tagging uses a **NASNet**-based TensorFlow model at 224×224 input, with labels loaded from a `labels.txt` file, and (in newer releases) an optional Ollama/vision-LLM integration as an alternative or supplement — [PhotoPrism label generation, search-derived](https://docs.photoprism.app/developer-guide/vision/label-generation/).
- Hardware acceleration: the ONNX face detector currently runs **CPU-only** — no GPU execution provider is wired up, throughput scales with thread count and host CPU; GPU support (NVIDIA CUDA on Linux, Apple CoreML on native macOS builds) is tracked as future work, not shipped — [PhotoPrism face recognition docs](https://docs.photoprism.app/developer-guide/vision/face-recognition/).
- Resource footprint: documented minimum is **2 CPU cores, 4 GB RAM**; TensorFlow-based features (RAW conversion, classification, face recognition) are explicitly **disabled below 1 GB RAM** — [PhotoPrism system requirements](https://www.photoprism.app/plus/kb/requirements).
- Database: PhotoPrism explicitly **recommends against SQLite for growing/large libraries**, steering users to MariaDB instead — SQLite is documented as suitable only for "small libraries and test environments where scalability and performance are not important" — [search-derived from PhotoPrism database docs](https://docs.photoprism.app/getting-started/advanced/databases/). This is a direct, relevant warning for LumenVault, whose architecture is SQLite-committed at 100k+ scale — though PhotoPrism's warning is about its general relational workload (metadata, search), not specifically about vector/embedding storage, and PhotoPrism doesn't use `sqlite-vec`-style extensions at all.
- **Implication for LumenVault**: PhotoPrism proves in-process, single-binary local ML is viable (no microservice required) — closer to LumenVault's architecture than Immich — but its own SQLite-scaling caution is a signal worth taking seriously and testing directly (with `sqlite-vec` + WAL mode + proper indexing) rather than assuming away.

## Open questions

- `sqlite-vec`'s own published 100k-vector benchmark (≤75ms/query at 768-dim) is a real primary-source number, but it's on the maintainer's hardware/data, not LumenVault's — should be reproduced on representative Windows hardware with actual CLIP/SigLIP embeddings before the schema is locked, not assumed to transfer exactly.
- Exact DirectML-provider `onnxruntime.dll` bundle size on Windows was not confirmed from a primary source (only the CPU-only core DLL's ~12 MB was confirmed) — matters for installer size budgeting.
- SigLIP2 per-checkpoint license was not individually re-verified for every size variant (only the SigLIP1-pattern of Apache-2.0 was directly confirmed) — worth a direct check on whichever specific checkpoint is eventually chosen.
- Whether `sqlite-vec`'s brute-force implementation itself uses SIMD internally, or whether a hand-rolled Rust SIMD scan (e.g. via `simsimd`) would meaningfully outperform it, was not confirmed from source — would need to be read from `sqlite-vec`'s C source directly.
- Immich's/PhotoPrism's resource-footprint numbers are for their *combined* app + ML workload, not isolated ML-only figures — not directly transferable to a LumenVault sizing estimate.
- Immich's model-provisioning gap (models fetched from Hugging Face on first use, not bundled in the image) means LumenVault's "ship weights in the installer, zero network ever" approach has no direct precedent to copy the packaging mechanics from — this needs its own design pass, not just a license clearance.
- OpenAI CLIP's and the LAION checkpoints' "deployed use is out of scope" model-card language is an ethics disclaimer, not a license clause (the MIT/Apache-2.0 grants themselves carry no legal use restriction) — but it should still get a legal read before shipping, since LumenVault would be exactly the kind of "deployed use" the disclaimer describes.
- Face grouping/recognition (InsightFace-style, as both precedents do) was out of scope for this pass per the ticket's framing around embeddings/tagging — flagged in MAP.md as a separate "face grouping yes/no" open question for the ML design ticket that follows this research. Note InsightFace's own license is *not* freely redistributable (see §4), unlike the CLIP/SigLIP family — that constraint alone may steer that future decision.
